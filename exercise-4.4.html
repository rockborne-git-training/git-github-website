4.3 Merge Conflicts

A merge conflict typically arise when two people have changed the same lines in a file or if one person deleted a file whilst another developer was working on it and Git doesn't know which version to keep.

Let's take a look at how to handle a merge conflict, but first we need to create one.

Decide between yourselves who's going to be Developer A and who's going to be Developer B.

Lesson Objectives

  • Understand what merge conflicts are and why they occur.
  • Learn how to identify merge conflicts in Git.
  • Practice resolving merge conflicts in GitHub.
  • Learn best practices for avoiding merge conflicts in collaborative projects.
Step 1 of 7

Make Sure You're in Different Branches

To create a merge conflict you need to make sure you're in different development branches.

It's important to note that you cannot create a merge conflict if two people are working on the same branch.

Git will not detect conflicts in the same branch and will simply take the latest committed change.

Developer A: Stay on the master branch, you do not need to do anything.

Developer B: Create a new branch called merge-conflict.

Run the following command in Git Bash

git checkout -b merge-conflict
Step 2 of 7

Edit the Same Line of Code

Now we're all going to create our own seperate branches.

Both Developer A and Developer B should edit line 25 of code.

  • Insert a pseudo landline number
  • Make sure the landline numbers you insert are not the same
  • Save the file (Ctrl + S)
Step 3 of 7

Stage, Commit and Push the Changes

Now we need to add the file to the staging index, commit it, and push to your respective branches.

Developer A: Run the following command in Git Bash

git push origin master

Developer B: Run the following command in Git Bash

git push origin merge-conflict
Step 3 of 7

Review the Merge on GitHub

We're now going to deal with the merge conflict on GitHub.

  • Go to your git-team-exercise repository on GitHub
  • Click on Compare & Pull Request
  • GitHub should state that it can't automatically merge the code, but we can still create the pull request
  • Scroll down and click Create pull request
Step 3 of 7

Manually Fixing the Merge Conflict in GitHub

We're now going to deal with the merge conflict on GitHub.

After we've clicked on Create pull request, GitHub will tell you that the branch has conflicts that must be resolved, as well as which file it occurs in.

Let's see how we can resolve this.

  • Click Resolve conflicts
  • The code which has the merge conflict in should appear on the page, and is specifically highlighted with yellow next to it

What do the symbols mean?

  • <<<<<<< branch-name: Everything below this line shows you what’s on the current branch
  • ======= : Is the end of the original lines, everything that follows is what’s on the branch that’s being merged in.
  • >>>>>>> branch-name: Ending indicator of what’s on the branch that’s being merged in
    • Step 3 of 7

    Choose Which Version to Keep

    GitHub has told you exactly where the conflict is in your code file.

    From here, you can either choose one version, combine both, or make your own change.

    For the purpose of exercise, we'll delete the version from the master branch to show the sequence of steps

    • Delete lines 25, 26 and 28
    • You should only be left with one landline number, mobile number and email inside the contact-details div
    Step 3 of 7

    Confirm the Merge

    Now that we've chosen which version to keep, we can mark the conflict has resolved and continue to merge the changes into the master branch.

    • Scroll back up the page and click Mark as resolved
    • Click Commit merge
    • The page should display that there are no conflicts with base branch because we've resolved it
    • Click Merge pull request
    • Click Confirm merge
    • Now you can delete the merge-conflict branch

    You have now successfully resolved your first merge conflict.

    Lesson Summary

  • Created a merge conflict by editing the same line of code in two different branches.
  • Reviewed the merge conflict on GitHub and identified the conflicting code.
  • Resolved the conflict by choosing which version of the code to keep.
  • Successfully merged the changes and deleted the conflict branch.
